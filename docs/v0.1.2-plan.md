## Scope

v0.1.2 is a focused performance release for the existing wgridder API.
The goal is to reduce avoidable hot-path work without changing the
mathematical operator, public plan-then-call pattern, sign convention,
or ducc parity targets.

The high-priority work is ordered by expected value; implementation
risk is recorded per-Part rather than driving the ordering:

1. Windowed forward accumulation without per-plane full-row scatter.
2. A no-w / constant-w fast path.
3. Plan-side precomputation of FINUFFT coordinates.
4. `w_strategy="auto"` using plan diagnostics.
5. GPU benchmark suite (GH200 target).
6. GPU benchmark sweep and cuFINUFFT-aware defaults.

Each item is independent enough to ship alone, but the first three are
implementation optimisations and should be benchmarked together before
changing defaults.

## Non-goals

- Do not change the w-kernel, `phi_hat` oversampling defaults, or
  `W_OVERSAMPLE_X0`.
- Do not change the public semantics of `dirty2vis`, `vis2dirty`, or
  `make_plan`.
- Do not promote a new default strategy until benchmark evidence covers
  the default telescope fixtures and at least one pathological
  w-distribution.
- Do not optimise CPU at the expense of breaking GPU traceability.

## Deprecation reminder

AGENTS.md §9 commits to removing the `"scan"` / `"vmap"` aliases in
v0.2. v0.1.2 is the last minor before that. The plan deliberately
does **not** escalate the deprecation (e.g. to `FutureWarning`) in
this release — agents picking this up should not add escalation
silently. If we change our mind, do it as an explicit Part of a
future plan.

## How to land this plan

This plan is written for an agent landing the work in small commits.
Rules:

1. **Every commit must leave the fast suite green.**
   `pixi run -e test pytest -q` must pass on every commit. If a step
   requires changing both code and tests, do them in the same commit.
   CI (`.github/workflows/test.yml`) runs on every push, so a red
   commit blocks the next push.
2. **One sub-step = one commit.** Each Part below has a "Sub-steps
   (one commit each)" section. Commit at every sub-step boundary
   even if the next sub-step is obvious — the boundaries are chosen
   so reverting any single commit leaves the repo working.
3. **Commit message format:** `v0.1.2 Part N.M: <subject>` for
   per-step commits; `v0.1.2: <subject>` for cross-cutting commits
   (baseline JSON, README, version bump).
4. **Order to land Parts:**
   - Part 0 (baseline) first — its commit IS the JSON file.
   - Parts 1, 2, 3 are independent and can land in any order.
   - **Exception:** if Part 3 picks option (c) — replacing
     `uvw_lambda` entirely — land Part 3 BEFORE Part 1 (or hold
     Part 1) to avoid touching `_channel_forward_windowed` twice.
   - Part 4 depends on Parts 1-3 having landed and benchmarked,
     because the heuristic gates on `n_w` and
     `window_padding_overhead`, which Part 2 may change for the
     constant-w case.
   - Part 5 (GH200 benchmark suite) is largely CPU-developable —
     the harness can be unit-tested on CPU; only the final baseline
     capture (sub-step 5.5) requires GH200 access. Land the
     harness early; it's a prerequisite for Part 6.
   - Part 6 is gated on GPU hardware availability AND Part 5 having
     landed — see Part 6's Fallback path. If GH200 access is
     unavailable, ship Parts 1-5 and follow Part 6's fallback.
5. **Don't squash the per-step commits at PR time.** The commit
   history is the audit trail for which sub-step introduced any
   regression.
6. **Plan-field changes invoke a checklist.** Any Part adding to
   `WGridderPlan` (Parts 2 and 3) must update — in the same commit
   — `_plan_aux`, the `register_pytree_node` flatten/unflatten, AND
   AGENTS.md §4's static-fields list. AGENTS.md §11 makes this a
   "don't" because the pytree registration is positional and silent
   to mismatches.

# Part 1: sorted-order windowed forward accumulation

## Problem

`_channel_forward_windowed` already sorts visibilities by `w` so that
each plane can process a contiguous window. However, after each plane's
NUFFT it scatters the window result into a fresh full-length
`(n_rows,)` vector in original row order, then adds that vector to the
scan accumulator.

This means every w-plane pays for:

- allocating / materialising a full-row zero vector;
- a scatter into original row order;
- a full-row add into the carry.

That cost is independent of the live window length and partly erases
the benefit of windowing.

## Proposed change

Keep the windowed forward accumulator in sorted row order:

1. `plane_to_sorted_update` applies
   `acc.at[lo : lo + max_window_size].add(contrib)` using
   `jax.lax.dynamic_update_slice` semantics.
2. The scan carry is a sorted-order visibility vector of shape
   `(n_rows,)`.
3. After the plane scan, unsort once:

```python
vis_original = jnp.empty_like(vis_sorted).at[plan.sort_perm].set(vis_sorted)
```

The vmap variant keeps the current full-row materialisation; the win
from sorted-order accumulation is specific to scan, where the carry
update is the hot loop.

## Clamping invariant (must preserve)

The existing windowed code clamps `lo` to
`[0, max(n_rows - max_window_size, 0)]` (`wgridder.py:201-205, 401-406`)
so that `dynamic_slice` is always in-bounds with a fixed slice size.
The sorted-order accumulator depends on the same invariant: every
plane's `.at[lo : lo + max_window_size].add(...)` must fit inside
the `(n_rows,)` accumulator. Carry that clamp into the new code
unchanged. Note that this means adjacent planes' slices **overlap**
in sorted order — that's the correct behaviour (a row inside multiple
planes' kernel support contributes from each).

## HLO/jaxpr sanity check

The carry-shape change should shrink the per-step HLO: the
`(n_rows,)` zero-allocation and gather/scatter to original order
disappear from the inner step and run once outside the scan. As a
pre-merge sanity step, eyeball the jaxpr or HLO of
`_dirty2vis_jit` for one of the bench fixtures before/after to
confirm the scan body got smaller, not larger. (`jax.make_jaxpr`
plus the bench plan is enough.)

## Correctness tests

Run:

```sh
pixi run -e test pytest tests/test_boundary_planes.py tests/test_against_dft.py \
  tests/test_adjoint.py tests/test_against_ducc.py -q
```

Specific checks:

- dense-vs-windowed forward parity remains within the existing
  tolerance (`test_boundary_planes.py`);
- boundary-plane tests still include rows exactly at kernel support
  edges;
- dot-product identity remains valid for `windowed_scan`
  (`test_adjoint.py::test_windowed_dot_product_identity` — this is
  the load-bearing test for Part 1, since a sorted-order
  accumulation bug would silently break
  `<forward(x), y> == <x, adjoint(y)>` while still producing
  plausible-looking outputs).

## Expected impact

Largest impact should be on `dirty2vis(..., w_strategy="windowed_scan")`
when `max_window_size << n_rows` and `n_w` is large. The change may also
make `windowed_scan` competitive for forward workloads where it is
currently held back by scatter overhead.

Risk: medium-low. The algorithm is unchanged, but row-order mistakes
are easy. Keep dense-vs-windowed parity tests tight.

# Part 2: no-w / constant-w fast path

## Problem

When all `w_lambda` values are identical, the current planner still
constructs at least `w_kernel_width + 1` planes. The operators then run
a stack of 2D NUFFTs even though the w-direction interpolation has
collapsed to a single phase state.

For coplanar / zenith-like data this can be an avoidable `~n_w`
multiplier.

## Proposed change

Add plan metadata describing w-degeneracy:

- `w_extent`: max-min over all channels in wavelengths;
- `is_constant_w`: a static bool, using exact zero for now or a
  documented tolerance derived from `epsilon` and `max|n-1|`;
- `w_constant`: the per-channel constant w-value (typically not zero
  — e.g. snapshot observations at constant pointing).

When `is_constant_w` is true:

- forward: multiply by `exp(+2πi w_constant (n-1)) / phi_hat_const`,
  run one `nufft2`, return — no per-row `phi(z)` weight;
- adjoint: run one `nufft1`, multiply by
  `exp(-2πi w_constant (n-1)) / phi_hat_const`, then apply the
  existing `1/n` output convention.

The simplest first version can trigger only when `w_extent == 0.0`.
That avoids tolerance semantics and still accelerates exact coplanar
test cases. A later patch can broaden it to "near-constant w" after a
parity sweep.

## Math subtleties to verify against ducc parity

Two details of the dense path do NOT carry over to the single-plane
fast path and will quietly bias results by a constant factor if
copied verbatim:

1. **W-plane centre is `w_constant`, not zero.** With a single plane
   the centre must equal the data's constant w-value so the residual
   phase `exp(2πi (w - w_k)(n-1))` is identically 1 across rows.
2. **No `(W/2)` factor in `phi_hat`.** The dense correction
   (`planning.py:305`) carries a `w_kernel_width / 2` multiplier that
   compensates the discretisation of the continuous w-integral into a
   sum over planes (`sum_k phi(z_k) ≈ (1/dw) ∫ phi(z) dw'`). With one
   plane there is no integral being discretised; the right correction
   is `phi_hat_const = phi_hat_table.evaluate(0.0)` — a scalar — and
   the `(W/2)` factor must be dropped.
3. **No per-row `phi(z)` kernel weight.** The kernel is unity at
   `z=0` by construction and there are no neighbouring planes
   contributing, so the weight is dropped entirely (not folded in).

Without (2) the fast path will appear to "work" but parity vs the
dense path or ducc will be off by exactly `W/2` (≈3 for `epsilon=1e-6`,
W=6) — large enough that the existing parity tests will catch it,
but it's worth writing the test that pins down the math, not just the
output magnitude.

## Plan-time vs call-time specialisation

Two reasonable implementations:

- **Plan-time specialisation (recommended).** When `is_constant_w` is
  true, `make_plan` sets `n_w=1`, `w_centers=[w_constant]`, rewrites
  `phi_hat_n` without the `(W/2)` factor, and skips the windowed
  builder. The JIT-compiled operator then runs the existing dense
  loop with `n_w=1` and no special branch.
- **Call-time branch.** The plan is built normally; the operators
  branch on the static `is_constant_w` flag and run a separate
  fast-path callable.

Plan-time is preferred because the static `n_w=1` lets the JIT prune
the dense-loop scan to a single iteration, and the `phi_hat_n` and
`w_centers` fields stay self-consistent (no "valid for n_w>1 only"
caveat baked into the dataclass). The call-time branch is simpler to
revert if it turns out to be a bad idea.

## Correctness tests

Add explicit tests with:

- all `uvw[:, 2] = 0`;
- all `uvw[:, 2] = constant_nonzero` (the case most likely to expose
  a missing `w_constant` in the w-plane centre);
- multiple channels with the same metre-space constant w, verifying
  per-channel wavelength scaling.

Plus a planning-level assertion:

- `is_constant_w == (w_extent == 0.0)` after `make_plan`;
- when `is_constant_w`, the plan has `n_w == 1` and `w_centers[0]`
  equals the data's constant w (per channel).

Compare fast path against:

- direct DFT on tiny problems;
- dense path on the same plan, if the implementation preserves a way
  to force the generic route in tests (e.g. an internal
  `_force_generic` planner kwarg used only by tests);
- ducc parity for at least one telescope-like coplanar fixture.

## Expected impact

For exact coplanar cases, expect roughly one NUFFT instead of
`w_kernel_width + 1` NUFFTs. For `epsilon=1e-6` (`W=6`) that is around
a 7x reduction in NUFFT calls, minus fixed overhead.

Risk: medium. The single-plane normalisation must match the existing
kernel/`phi_hat_n` convention exactly.

## README update

Document the fast path in the README "Performance notes" section:
which inputs trigger it (`w_extent == 0`), the expected speedup, and
the fact that `plan.n_w == 1` is the user-visible signal that the
specialisation engaged.

# Part 3: precompute FINUFFT coordinates in the plan

## Problem

Every channel helper recomputes:

```python
u_ft = (2*pi * plan.pixsize_l) * uvw[..., 0]
v_ft = (2*pi * plan.pixsize_m) * uvw[..., 1]
```

These arrays depend only on the plan. The dense and windowed paths do
this separately for forward and adjoint.

## Proposed change

Three implementations, in order of increasing aggressiveness:

(a) **Add both dense and sorted FINUFFT coords:**

- `u_finufft`: `(n_chan, n_rows)`
- `v_finufft`: `(n_chan, n_rows)`
- `u_finufft_sorted`: `(n_chan, n_rows)`
- `v_finufft_sorted`: `(n_chan, n_rows)`

Cheapest to land, most plan memory. Doubles the coord-side memory at
telescope scale (already `(n_chan, n_rows, 3)` floats for the two
uvw_lambda arrays — adding four more flat arrays of the same shape
roughly **doubles total plan memory**).

(b) **Add dense FINUFFT coords only, gather at scan time for windowed.**
Keeps `uvw_lambda_sorted` but skips `u/v_finufft_sorted`; the
windowed helpers do `dynamic_slice(u_finufft, …)` then a sort-perm
gather. Saves half the new memory at the cost of one extra gather
per scan iter. Probably the right default.

(c) **Replace `uvw_lambda` and `uvw_lambda_sorted` with separate
`u/v/w_finufft[_sorted]` arrays.** Net memory unchanged from today
(same total floats, different shape); per-call arithmetic drops
because the `2π * pixsize` scaling is baked into `u/v_finufft`.
Largest blast radius — every caller of `plan.uvw_lambda*` must be
updated — but the cleanest end state.

Pick (b) if landing fast matters; (c) if v0.1.2 also wants to
simplify downstream code. Decide before writing tests, since the
pytree leaf count differs between options.

## Correctness tests

- Update pytree leaf count tests deliberately — this is a real
  pytree-shape change, not a fix.
- Assert coordinates match the old formula in `test_planning.py`
  (`u_finufft[c, r] == 2π · pixsize_l · uvw_lambda[c, r, 0]`).
- Confirm `_plan_aux` and the `register_pytree_node` flatten/unflatten
  in `planning.py` stay in sync with the new leaf list (see AGENTS.md
  §4 invariant).
- Run the full default suite.

## Expected impact

Small but low-risk reduction in per-call arithmetic and HLO size. This
is most useful because it also simplifies later fast paths.

Risk: low, apart from plan-size growth and pytree aux/leaf drift.

## Sub-steps (one commit each)

adapt if (a) or (c) is picked.

# Part 4: `w_strategy="auto"`

## Problem

Users currently need to know when dense vs windowed strategies win.
The plan already exposes enough diagnostics to make a conservative
choice automatically.

## Proposed heuristic

Start with a CPU-safe heuristic:

```python
def _auto_w_strategy(plan, *, is_adjoint: bool) -> str:
    # TODO(v0.1.2 Part 6): make this platform-aware via
    # jax.devices()[0].platform once the GH200 sweep lands. The CPU
    # gates below come from AGENTS.md §9 measured wins: windowed
    # only beats dense on the adjoint at off-zenith, with no
    # measurable forward win.
    if plan.n_w <= plan.w_kernel_width + 2:
        return "dense_scan"
    if plan.window_padding_overhead > 5.0:
        return "dense_scan"
    if is_adjoint and plan.n_w / plan.w_kernel_width > 2.0:
        return "windowed_scan"
    return "dense_scan"
```

The forward operator deliberately never picks windowed, because v0.1.1
measurements found windowed forward roughly flat vs dense (AGENTS.md
§9). If Part 1 (sorted-order forward) changes that, revisit this
heuristic in the same PR.

Add `"auto"` as an accepted public value for `w_strategy`, but do not
make it the default until the benchmark suite confirms it does not
choose a loser on the standard telescope matrix.

## Resolution must happen before the JIT boundary

`w_strategy` is declared in `static_argnames` on both
`_dirty2vis_jit` and `_vis2dirty_jit` (`wgridder.py:283, 438`). If
the literal string `"auto"` reaches the JIT entry point it becomes
its own cache key, defeating the point: identical plans calling with
`"auto"` would never share a compiled binary with the same plan
calling with the resolved canonical name.

The fix is to resolve `"auto"` **inside the public wrapper**,
alongside `_canonicalise_w_strategy`, so the static arg passed into
JIT is always one of the four canonical names:

```python
def _canonicalise_w_strategy(name, *, plan=None, is_adjoint=None):
    if name == "auto":
        if plan is None or is_adjoint is None:
            raise ValueError("auto requires plan + is_adjoint context")
        return _auto_w_strategy(plan, is_adjoint=is_adjoint)
    # ...existing canonical / deprecated-alias handling
```

`dirty2vis` and `vis2dirty` call this with their respective
`is_adjoint` flags. Inside JIT, `w_strategy` is always canonical and
the cache works as it does today.

Note: this means two different plans (with different `n_w` /
`window_padding_overhead`) can resolve `"auto"` to different canonical
names — that's intentional and is the same compile-once-per-plan
behaviour users get if they pick the strategy explicitly.

## Correctness tests

- `_canonicalise_w_strategy("auto")` should not emit a deprecation
  warning.
- `dirty2vis(..., w_strategy="auto")` and `vis2dirty(...,
  w_strategy="auto")` should match the explicitly selected strategy.
- Existing deprecated aliases `"scan"` and `"vmap"` must keep their
  current warnings.

## Benchmark acceptance

For each telescope / pointing in the benchmark matrix, the auto-picked
strategy should be within 15% of the best CPU strategy for the same
operator (matching Part 6's GPU acceptance bar), or intentionally
choose dense because the faster strategy is too memory-heavy.

Expected impact: 1.2-1.5x adjoint speedup on off-zenith workloads for
users who otherwise stay on the default.

Risk: low while `"auto"` is opt-in.

## README update

Add `"auto"` to the documented `w_strategy` values and a short
example showing the recommended pattern (`w_strategy="auto"` for
new code; explicit names when reproducing benchmarks).

## Sub-steps (one commit each)

**4.1 Add `_auto_w_strategy` helper (private, unit-tested).**
*Files:* `src/jax_nufft/wgridder.py`, `tests/test_wgridder.py` (or
new `test_auto_strategy.py`).
Implement the heuristic above; unit-test it with synthetic plans
covering each branch (small `n_w`, large padding overhead,
adjoint-with-large-n_w, forward).
*Acceptance:* helper covered by deterministic unit tests; not yet
wired into the public API.

**4.2 Extend `_canonicalise_w_strategy` to accept `"auto"`.**
*Files:* `src/jax_nufft/wgridder.py`, `src/jax_nufft/_types.py`
(add `"auto"` to the `WStrategy` Literal alias).
Resolve `"auto"` to a canonical name using `plan` + `is_adjoint`
before the JIT boundary, exactly as described in the
"Resolution must happen before the JIT boundary" section above.
*Acceptance:* `dirty2vis(..., w_strategy="auto")` and the
explicitly-resolved equivalent produce bit-equal outputs (one new
test) and share a JIT cache entry (assert via `_dirty2vis_jit._cache_size()`
or by observing no recompile on the second call).

**4.3 README update + opt-in note.** One commit.
*Acceptance:* `"auto"` is documented but NOT promoted as default.

# Part 5: GPU benchmark suite (GH200 target)

## Problem

The existing benchmark suite (`tests/test_benchmark_against_ducc.py`)
is built around `pytest-benchmark` and a CPU mental model. Several
properties of that setup are wrong for GPU work:

- **No async sync.** JAX dispatch is non-blocking on GPU; without an
  explicit `.block_until_ready()` inside the timed callable,
  `pytest-benchmark` measures kernel-launch latency, not kernel
  completion. Timings look optimistic by 10–1000×.
- **No warmup separation.** First-call cost includes XLA compile +
  cuFINUFFT plan creation, which is irrelevant to the steady-state
  optimisation-loop regime AGENTS.md §7 cares about. The harness
  must run warmup iterations outside the timer.
- **ducc-only baseline.** ducc is CPU-only, so on GH200 the
  natural comparisons are (a) JAX-on-GPU vs ducc-on-CPU (strategic
  value question) and (b) JAX-on-GPU vs JAX-on-CPU (what does the
  GPU port buy you). The current suite only does (a).
- **No GPU memory metric.** Peak RSS via `tracemalloc` (the current
  memory test pattern) doesn't see HBM allocations. Need
  `nvidia-smi` polling or `jax.profiler.device_memory_profile`.
- **No hardware fingerprint.** Comparing GH200 numbers to A100 or
  H100 numbers later requires an explicit fingerprint in the result
  JSON.

## GH200-specific considerations

The target hardware is NVIDIA GH200 (Grace Hopper Superchip):

- **Unified memory + NVLink-C2C.** Host↔device transfers run over
  900 GB/s NVLink-C2C rather than PCIe. Plan arrays sized by
  `(n_chan, n_rows, 3)` floats that would be a host↔device burden
  on A100 are essentially free here. Worth recording a "host
  placement" variant of the bench to show this.
- **96 GB HBM3 per GPU.** Large enough to bench MeerKAT-class plans
  end-to-end. Add a "large-plan" fixture that A100-40GB cannot
  hold; this is the case the GPU port is supposed to unlock.
- **ARM CPU (Grace, 72 cores).** Affects ducc CPU-side comparisons —
  ducc's threading defaults are tuned for x86. Pin `OMP_NUM_THREADS`
  explicitly and record it in the fingerprint so future re-benches
  are reproducible.
- **Single GPU.** Don't design the suite around multi-GPU (separate
  scope); just ensure it doesn't accidentally use `jax.pmap`.

## Proposed change

Add a new test module `tests/test_benchmark_gpu.py` plus a small
timing-harness module that the CPU benchmarks can also adopt later.

**Timing harness** (`tests/bench_harness.py`, new):

```python
def time_jax_callable(
    fn, *args,
    warmup: int = 3,
    iters: int = 20,
    sync: bool = True,
):
    """Time a JAX-returning callable with proper async sync.

    Returns dict with median, p05, p95, cv, iters. Warmup runs are
    not counted. With sync=True (GPU default), calls
    jax.block_until_ready on the output of every iteration.
    """
```

**GPU benchmark module** (`tests/test_benchmark_gpu.py`, new):

- Gated by `--runbench-gpu` (new pytest option in `conftest.py`) AND
  `jax.devices()[0].platform == "gpu"`. If either is missing, skip.
- Parametrises over:
  - `op`: `dirty2vis`, `vis2dirty`
  - `w_strategy`: all four canonical names
  - `channel_strategy`: `scan`, `vmap`
  - `fixture`: `short_telescope_pointing`, `long_telescope_pointing`,
    a new `gh200_large_pointing` (sized to use ~50 GB HBM)
- For each cell, records: median latency, p05/p95, CV across `iters`,
  peak HBM via `jax.profiler.device_memory_profile`, compile time
  (first-call latency minus steady-state median).
- Writes a single JSON to the path in the `JAX_NUFFT_BENCH_OUTPUT`
  env var (or a default under `/tmp` if unset) — a session-scoped
  fixture collects rows and dumps at teardown. Env var was chosen
  over a pytest CLI flag to keep `conftest.py` minimal and let CI
  scripts set the path without parsing pytest args.

**Hardware fingerprint** (`tests/bench_harness.py`):

A `capture_fingerprint()` function that writes a dict including:

- `nvidia-smi -L` output, parsed.
- `jax.devices()` repr.
- `jax.__version__`.
- `jax_finufft.__version__`. If the underlying cuFINUFFT C library
  version is exposed via a stable attribute, include it; otherwise
  drop it rather than reach into private modules.
- CUDA driver + version, via
  `nvidia-smi --query-gpu=driver_version --format=csv,noheader`
  (parse the first line). Stick to `nvidia-smi` output rather than
  `jax.lib.*` internals — those move between releases.
- `OMP_NUM_THREADS`, `XLA_FLAGS` env.
- Hostname (optional; helps when comparing internal vs external
  runs).

Every result JSON embeds this dict under a top-level `fingerprint`
key.

# Lower-priority follow-ups

These remain useful but should not block v0.1.2:

- **Reduce plan memory.** `uvw_lambda_sorted` duplicates coordinate
  storage. Worth revisiting if large multi-channel plans become a real
  pressure point.
- **Plan-time pre-compilation cache.** Useful for optimisation
  pipelines that want to pay compile cost during setup, but it does
  not improve steady-state runtime.
- **Per-channel `n_w`.** Potentially valuable for wideband
  observations, but it complicates static shapes, plan structure, and
  JIT cache behaviour.
- **Precomputed w-phase tables.** Could save repeated exponentials but
  costs `O(n_w * image_size)` plan memory. Benchmark before adding.
