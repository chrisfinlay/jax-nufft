# `jax-nufft` v0.1.1 plan

## Scope

Two independent performance improvements over v0.1 that, applied
together, should put CPU off-zenith timings within `~2-3x` of
`ducc0.wgridder` for the FFT-and-spread-dominated regimes. Public
API is unchanged. Algorithmic correctness against ducc and the direct
DFT is unchanged.

1. **Part 1 — standard `n_w`.** v0.1 oversampled the w-direction by a
   factor of `W/4` due to a safety choice on the kernel-correction
   range. Reverting to a `W`-independent oversampling (matching ducc)
   removes that factor without touching the algorithm.

2. **Part 2 — windowed per-plane scan.** v0.1 evaluates every
   visibility on every w-plane and relies on the kernel zeroing out
   non-contributing rows. Sorting visibilities by `w_lambda` makes the
   contributing set per plane a contiguous slice; using `dynamic_slice`
   with a fixed worst-case window keeps shapes static while cutting
   spread work to within `~2x` of ducc.

The v0.1 dense strategies (`scan`, `vmap`) are kept and renamed
(`dense_scan`, `dense_vmap`) as a reference path — primarily for
boundary-plane verification, where the windowed scheme's edge handling
diverges most from the dense baseline.

## Versioning

- Tag: `v0.1.1`
- Branch: `v0.1.1`
- The plan can be split: Part 1 alone is shippable as a pure planning
  fix if Part 2 needs more iteration. In that case Part 1 ships as
  `v0.1.1`, Part 2 follows as `v0.1.2`.

# Part 1: restore standard `n_w`

## What v0.1 does

In `planning.py`, v0.1 chooses

    x_0 = 1 / W
    dw  = x_0 / max|n - 1| = 1 / (W * max|n - 1|)
    n_w = ceil(w_extent / dw) + W = W * w_extent * max|n - 1| + W

The constant `W_OVERSAMPLE_ETA_MAX = 0.5` enforces this: it pins the
maximum value of `eta = (n - 1) * dw * (W / 2)` at 0.5 regardless of
`W`, by scaling `dw` accordingly.

This was chosen during v0.1 because `phi_hat(eta)` was observed to
become small enough at `eta > 0.5` that `1 / phi_hat` amplified
aliasing error and broke parity tests. The fix kept correctness but
inflated `n_w` linearly with `W`.

## What ducc does (to verify)

ducc uses a `W`-independent oversampling for the w-direction — i.e.
`x_0` does not depend on `W`. The v0.2 plan claimed `x_0 = 0.25` but
this is to be **verified empirically**, not taken as canonical. The
textbook Beatty-style value for `sigma = 2` is `x_0 = 0.5`, and any
deviation in ducc reflects implementation-specific kernel tuning.

The verification step in implementation:

1. Inspect `ducc/src/ducc0/wgridder/wgridder.h` (read-only) for the
   formula it uses to choose `nu/nv` and the w-direction analog. Note
   the exact relationship between `epsilon`, kernel parameters, and
   the oversampling ratio.
2. Run a small parity sweep: for `W in {4, 6, 8, 10}` and `eps in
   {1e-4, 1e-6, 1e-8}`, measure ducc's effective `n_w` for a fixed
   problem (extract via timing or by reading ducc's plan structure if
   accessible from Python). Match `jax-nufft`'s `n_w` to within `+/-W`.
3. Pick `W_OVERSAMPLE_X0` to match. Likely either 0.25 or 0.5; commit
   to whichever ducc actually uses.

## What `phi_hat` needs at the wider eta range

The new `eta_max = W * x_0 / 2`. For `x_0 = 0.25` and `W = 8`,
`eta_max = 1.0`. For `x_0 = 0.5` and `W = 8`, `eta_max = 2.0`.

`phi_hat` is the FT of the exp-of-semicircle kernel and decays
super-exponentially. At `eta = 1` and `beta = 2.30 * W = 18.4` (W=8),
`phi_hat` is small but well above the safety floor. At `eta = 2` it's
much smaller and may cause `1 / phi_hat` to amplify error.

Two mitigations:

1. **Bump `phi_hat_oversample`** so the cubic-Lagrange interpolator
   resolves `phi_hat` accurately at the new range. Suggested defaults:

   | W      | current | new |
   |--------|---------|-----|
   | 4      | 32      | 32  |
   | 6      | 32      | 64  |
   | 8      | 32      | 64  |
   | >= 10  | 32      | 128 |

   Auto-select in `make_plan` based on `W`.

2. **`phi_hat` safety floor.** The current safety check asserts
   `min(phi_hat) > floor`. Document this is a *correctness* check, not
   a performance one — if it fails, the answer is wrong. Raise an
   informative error suggesting either (a) tighter `epsilon` /
   different `W`, or (b) falling back to v0.1's `x_0 = 1/W` mode for
   the offending configuration.

## Implementation steps

1. In `planning.py`:
   - Replace `W_OVERSAMPLE_ETA_MAX = 0.5` with a constant `W_OVERSAMPLE_X0`.
   - The value (0.25 vs 0.5) is set after the empirical ducc check above.
   - Recompute `dw`, `n_w`, and the resulting `eta_max_request` from
     `W_OVERSAMPLE_X0` instead of pinning `eta_max` to 0.5.
2. In `kernel.py`:
   - `phi_hat_oversample` becomes a function of `W` (or a `dict[int, int]`
     lookup), not a constant.
   - Update the `phi_hat` builder to accept the new range.
3. In tests:
   - The unit test that asserts `max_nm1 * w_kernel_scale <= 0.5`
     becomes `<= W_OVERSAMPLE_X0 * W / 2`.
   - Add a `phi_hat` conditioning test: for each `(W, eps)` pair in the
     test matrix, assert `min(phi_hat over the supported eta range) >
     safety_floor` and that cubic-Lagrange interp error is `< eps / 10`.
4. Re-run the existing ducc parity tests at all telescopes and
   epsilons. The expected outcome is identical accuracy with smaller
   `n_w`. If any fall below the `10 * eps` parity bound, debug
   `phi_hat` interpolation or kernel parameter choice before pushing.

## Expected speedup

FFT count is `n_w` per call. The reduction factor is `W / 4` (if ducc
uses `x_0 = 0.25`) or `W / 2` (if ducc uses `x_0 = 0.5`).

Assuming the more conservative `W / 4`:

| eps    | W | speedup factor |
|--------|---|----------------|
| 1e-4   | 4 | 1.0x           |
| 1e-6   | 6 | 1.5x           |
| 1e-8   | 8 | 2.0x           |
| 1e-10  | 10| 2.5x           |

Applied to MWA-extended off-zenith forward at `eps = 1e-6` (v0.1
benchmark: 818 ms scan, 586 ms vmap), expect `~545 ms / 390 ms`.

# Part 2: windowed per-plane scan

## Goal

Each visibility contributes to exactly `W` consecutive w-planes via the
gridding kernel. v0.1 pays the spread cost on every plane for every
visibility, with the kernel zeroing out the non-contributing
contributions after the fact. Restructuring around the kernel's
compact support — using only the visibilities that actually contribute
to a given plane — cuts spread work from `O(n_w * n_rows * W^2)` down
to `O(n_rows * W^3)` (within a worst-case-padding factor).

## Why sort by w

Without sorting, "visibilities contributing to plane `k`" is an
unstructured subset of the visibility array. Selecting them requires
either a boolean mask (which JAX handles via gather, breaking
contiguity) or an explicit scatter (which has its own overhead).

After sorting visibilities by `w_lambda`, the contributing set for
plane `k` is a **contiguous slice** in the sorted array. This:

- Eliminates the scatter — extraction is `lax.dynamic_slice`.
- Gives coalesced memory access on both CPU and GPU.
- Has static slice size (a JAX requirement), with the start index
  varying per plane.

The slice size is fixed to the worst-case window across all planes;
visibilities inside the slice but outside the kernel support get
weight zero from `phi(w_lambda - w_k)` and contribute nothing.

The sort happens once at plan time, on `w_meters` (frequency-
independent). The same permutation applies to all channels, but the
per-plane slice indices are per-channel because `w_lambda` ranges
differ.

## Plan-side additions

Augment `WGridderPlan` with:

| field             | shape                       | contents                                                |
|-------------------|-----------------------------|---------------------------------------------------------|
| `sort_perm`       | `(n_rows,)` int             | argsort of `uvw[:, 2]` (ascending)                       |
| `window_start`    | `(n_chan, n_w)` int         | start index in sorted array for plane k's window         |
| `window_size`     | `(n_chan, n_w)` int         | live (unpadded) window length                            |
| `max_window_size` | `int` (static)              | `max(window_size)` over all (chan, plane) pairs          |
| `window_padding_overhead` | `float` (static)    | `max_window_size / mean_window_size`, for diagnostics    |

Compute these in `make_plan` with:

```python
sort_perm = jnp.argsort(uvw[:, 2])
uvw_sorted = uvw[sort_perm]

for c in range(n_chan):
    w_lambda_sorted = uvw_sorted[:, 2] * freq[c] / C
    half_W_dw = (W / 2) * dw[c]
    for k in range(n_w[c]):
        w_k = w_centers[c, k]
        lo = jnp.searchsorted(w_lambda_sorted, w_k - half_W_dw)
        hi = jnp.searchsorted(w_lambda_sorted, w_k + half_W_dw)
        window_start[c, k] = lo
        window_size[c, k]  = hi - lo

max_window_size = int(window_size.max())
mean_window_size = float(window_size.mean())
window_padding_overhead = max_window_size / mean_window_size
```

`searchsorted` is non-differentiable but it's only used for indexing,
not in the gradient path.

If `window_padding_overhead > 3.0`, raise a warning suggesting the
caller fall back to `dense_scan` (the w-distribution is pathological,
likely a clumped UV pattern that defeats the windowing assumption).
If `> 5.0`, raise an error unless `force_windowed=True`.

## Forward operator

```python
def dirty2vis_windowed(plan, image):
    nm1 = plan.n_minus_1
    phi_hat_n = plan.phi_hat_n  # (n_l, n_m)

    def per_chan(c):
        # Pre-permuted FINUFFT coords (stored in plan in sorted order)
        u_sorted = plan.u_finufft_sorted[c]
        v_sorted = plan.v_finufft_sorted[c]
        w_lambda_sorted = plan.w_lambda_sorted[c]

        def per_plane(carry, k):
            lo = plan.window_start[c, k]
            size = plan.window_size[c, k]
            w_k = plan.w_centers[c, k]

            # rigid-size slice
            u_k = jax.lax.dynamic_slice(u_sorted, (lo,), (plan.max_window_size,))
            v_k = jax.lax.dynamic_slice(v_sorted, (lo,), (plan.max_window_size,))
            w_k_lambda = jax.lax.dynamic_slice(w_lambda_sorted, (lo,), (plan.max_window_size,))

            # mask for live (unpadded) entries
            mask = jnp.arange(plan.max_window_size) < size

            # image-domain corrections for this plane
            image_k = image[c] * jnp.exp(2j * jnp.pi * w_k * nm1) / phi_hat_n

            # 2D NUFFT type 2: image -> non-uniform points
            contrib = jax_finufft.nufft2(image_k, u_k, v_k, iflag=-1, eps=plan.eps)

            # apply w-direction kernel + mask
            z = (w_k_lambda - w_k) / plan.w_kernel_scale[c]
            phi_w = phi_kernel(z, plan.beta)
            contrib = contrib * phi_w * mask

            # scatter back to original visibility order
            rows_k = jax.lax.dynamic_slice(plan.sort_perm, (lo,), (plan.max_window_size,))
            vis_acc = carry.at[rows_k].add(contrib)
            return vis_acc, None

        vis_init = jnp.zeros((plan.n_rows,), dtype=image.dtype)
        vis_c, _ = jax.lax.scan(per_plane, vis_init, jnp.arange(plan.n_w[c]))
        return vis_c

    # vmap or scan over channels — same options as v0.1
    if plan.channel_strategy == "vmap":
        vis = jax.vmap(per_chan)(jnp.arange(plan.n_chan)).T  # (n_rows, n_chan)
    else:
        vis = jax.lax.scan(lambda _, c: (None, per_chan(c)), None, jnp.arange(plan.n_chan))[1].T

    return vis
```

The scatter `carry.at[rows_k].add(contrib)` lowers to an XLA scatter-add.
Padded entries in `rows_k` are valid indices (they refer to real rows
in the original ordering), so the scatter is in-bounds; their contrib
is zero from the mask, so they contribute nothing.

## Adjoint operator

Mirror structure:

```python
def vis2dirty_windowed(plan, vis):
    nm1 = plan.n_minus_1
    phi_hat_n = plan.phi_hat_n
    n = nm1 + 1.0

    def per_chan(c):
        u_sorted = plan.u_finufft_sorted[c]
        v_sorted = plan.v_finufft_sorted[c]
        w_lambda_sorted = plan.w_lambda_sorted[c]

        # vis is in original order; gather sorted version
        vis_sorted = vis[:, c][plan.sort_perm]
        if plan.weights is not None:
            weights_sorted = plan.weights[:, c][plan.sort_perm]
        else:
            weights_sorted = None

        def per_plane(carry, k):
            lo = plan.window_start[c, k]
            size = plan.window_size[c, k]
            w_k = plan.w_centers[c, k]

            u_k = jax.lax.dynamic_slice(u_sorted, (lo,), (plan.max_window_size,))
            v_k = jax.lax.dynamic_slice(v_sorted, (lo,), (plan.max_window_size,))
            w_k_lambda = jax.lax.dynamic_slice(w_lambda_sorted, (lo,), (plan.max_window_size,))

            mask = jnp.arange(plan.max_window_size) < size

            vis_k = jax.lax.dynamic_slice(vis_sorted, (lo,), (plan.max_window_size,)) * mask
            if weights_sorted is not None:
                w_k_arr = jax.lax.dynamic_slice(weights_sorted, (lo,), (plan.max_window_size,))
                vis_k = vis_k * w_k_arr

            # apply w-direction kernel
            z = (w_k_lambda - w_k) / plan.w_kernel_scale[c]
            vis_k = vis_k * phi_kernel(z, plan.beta)

            # 2D NUFFT type 1: non-uniform -> image
            H_k = jax_finufft.nufft1((plan.n_l, plan.n_m), vis_k, u_k, v_k, iflag=+1, eps=plan.eps)

            # image-domain correction and accumulate
            I_k = H_k * jnp.exp(-2j * jnp.pi * w_k * nm1) / phi_hat_n
            return carry + I_k, None

        dirty_init = jnp.zeros((plan.n_l, plan.n_m), dtype=jnp.complex64 if vis.dtype == jnp.complex64 else jnp.complex128)
        dirty_c, _ = jax.lax.scan(per_plane, dirty_init, jnp.arange(plan.n_w[c]))
        return (dirty_c.real / n)

    # vmap or scan over channels — same options as v0.1
    ...
```

## Cost analysis

Define `mean_window_size = n_rows * W / n_w` (every visibility appears
in `W` planes, total plane-touches `n_rows * W`, distributed across
`n_w` planes).

Let `p = max_window_size / mean_window_size` be the padding overhead;
`p ~ 1.5-2` for reasonably uniform w-distributions, `p ~ 3-5` for
clumped ones.

### Spread cost (per call)

| Strategy        | Spread work                                    |
|-----------------|------------------------------------------------|
| v0.1 dense      | `n_w * n_rows * W^2`                           |
| v0.1.1 windowed | `n_w * (p * mean_window_size) * W^2 = p * n_rows * W^3` |
| ducc            | `n_rows * W^3`                                 |

So v0.1.1 windowed is `p ~ 2x` ducc and `n_w / (p * W) ~ n_w / 16x`
faster than v0.1 dense.

### FFT cost (per call)

Unchanged by Part 2 — still `n_w` 2D FFTs per channel. Part 1 reduces
`n_w` by `W / 4` (or `W / 2`), giving the FFT-side improvement.

### Combined (Parts 1 + 2)

| Term   | v0.1                          | v0.1.1                       | vs ducc  |
|--------|-------------------------------|------------------------------|----------|
| FFT    | `W * w_extent * max\|n-1\|`   | `4 * w_extent * max\|n-1\|`  | matches  |
| Spread | `n_w * n_rows * W^2`          | `p * n_rows * W^3`           | `~p`x    |

## Edge planes

The first and last `~W/2` planes have one-sided kernel support:
half the kernel window falls outside `[w_min, w_max]`. Two issues:

1. `searchsorted(w_k - half_W_dw)` may return 0 (empty left side).
   `searchsorted(w_k + half_W_dw)` may return `n_rows` (empty right
   side). Both are correctly handled by `dynamic_slice` provided
   `lo + max_window_size <= n_rows` is enforced — clamp `lo` to
   `[0, n_rows - max_window_size]`.
2. Edge planes have smaller `window_size`, so most of the
   `max_window_size` slice is masked out. This wastes work proportional
   to `(W/2) / n_w` of total work; negligible for typical `n_w >> W`.

The `dense_scan` reference path is the comparison target for edge
plane behavior. The windowed and dense paths must agree to within
floating-point round-off on a parity test that runs only the edge
planes (e.g., a synthetic w-distribution with all visibilities clumped
near `w_min`).

## Why not bucketed scan

A bucketed alternative (pre-partition visibilities into `B` rigid-size
buckets each spanning `M * dw` of w-space, iterate over buckets
outer-loop, then over the `M + W` planes each bucket overlaps inner
loop) was considered and rejected because it doubles the NUFFT count.

Each bucket contributes to its `M` interior planes plus `~W` boundary
planes from neighbouring buckets' kernel reach. For `M = W` (the
natural choice), total NUFFTs become `B * (M + W) = (n_w / W) * 2W =
2 * n_w` — twice the windowed scheme. The spread *cost* per call is
comparable, so the doubled NUFFT count means doubled FFT work. CPU
loses; GPU is unclear without measurement.

The windowed scheme has **exactly `n_w` NUFFT calls per channel**, the
minimum possible without changing the algorithmic decomposition. It is
strictly preferred until GPU benchmarks demonstrate otherwise. Bucketed
scan is not implemented in v0.1.1.

# Reference path: keeping v0.1 dense strategies

Rename existing strategies and keep them available:

| v0.1 name | v0.1.1 name   | role                                       |
|-----------|---------------|--------------------------------------------|
| `scan`    | `dense_scan`  | reference, boundary-plane parity, fallback |
| `vmap`    | `dense_vmap`  | reference, fallback                        |
| —         | `windowed_scan` (new, default) | production path             |
| —         | `windowed_vmap` (new, optional) | high-memory variant        |

The `dense_*` strategies remain documented and tested. They are the
correctness baseline for the windowed strategies. They are also the
recommended fallback when:

- `window_padding_overhead > 5.0`
- The w-distribution is clumped enough that windowing wastes more work
  than it saves
- A user wants to reproduce v0.1 results

## Default strategy

`windowed_scan` becomes the default `w_strategy` if it passes parity
and benchmark gates. If benchmark wins are inconsistent (e.g., wins
on MWA-extended but loses on EDA2 because `n_w = 1`), keep `dense_scan`
as the default and have the user opt in to `windowed_scan`. A small
heuristic in `make_plan` could auto-select: `windowed_scan` when
`n_w / W > 2`, `dense_scan` otherwise. Decide based on the benchmark
sweep outcome.

## Backwards compatibility

For one release: accept the v0.1 strategy names (`scan`, `vmap`) as
deprecated aliases that map to (`dense_scan`, `dense_vmap`) with a
`DeprecationWarning`. Remove in v0.2.

# Multi-channel handling

The sort key is `w_meters` (frequency-independent), so a single
`sort_perm` of shape `(n_rows,)` serves all channels. Per-channel
quantities are:

- `n_w[c]`, `dw[c]`, `w_kernel_scale[c]`, `w_centers[c]` — already
  per-channel in v0.1.
- `window_start[c, k]`, `window_size[c, k]` — new in v0.1.1, shape
  `(n_chan, n_w[c])`.
- `max_window_size` — single static int, taken over all channels and
  planes.

Memory cost of the new tables is `n_chan * n_w * 8 bytes`. For
MeerKAT-class problems with `n_chan = 1024` and `n_w = 800`, that's
6.5 MB — negligible.

The pre-permutation of `(u_finufft, v_finufft, w_lambda)` into sorted
order is done once at plan time and stored in the plan. This trades
~2x memory on coords (cheap, `(n_rows, n_chan)` floats) for one fewer
gather per scan iteration. Recommended; the alternative (apply
`sort_perm` at scan time per channel) is fine but slightly slower.

The scatter at the end of forward (`carry.at[rows_k].add(contrib)`)
needs `rows_k` in original order — extracted from `plan.sort_perm`
via `dynamic_slice`, same start index `lo`.

# Testing

## New tests

1. **Window-builder tests** (`tests/test_planning.py`):
   - `window_size.sum(axis=1)` over planes equals `n_rows * W` modulo
     edge effects.
   - `window_start[c, k]` is monotonic in `k`.
   - All `window_start[c, k] + max_window_size <= n_rows` (clamping
     check).
   - Padding overhead is computed correctly for synthetic uniform and
     clumped w-distributions.

2. **Windowed-vs-dense parity tests** (`tests/test_strategies.py`):
   - For all four telescopes x both pointings x `eps in {1e-4, 1e-6,
     1e-8}`: `dirty2vis(windowed_scan)` matches `dirty2vis(dense_scan)`
     to within `eps` (relative L2).
   - Same for `vis2dirty`.

3. **Boundary-plane stress tests** (`tests/test_boundary_planes.py`):
   - Synthetic w-distribution with all visibilities clumped near
     `w_min` (only edge planes have non-trivial windows).
   - Same near `w_max`.
   - Symmetric distribution (most contribution near zero).
   - Bimodal distribution (clumps at both ends).
   - Each must give windowed-vs-dense agreement to within `eps`.

4. **`phi_hat` conditioning tests** (`tests/test_kernel.py`):
   - For `W in {4, 6, 8, 10}`:
     - `min(phi_hat over [-eta_max, eta_max]) > safety_floor`.
     - Cubic-Lagrange interp error vs reference dense FT `< eps / 10`.

5. **`n_w <= W` regression test**:
   - For EDA2 zenith (where `n_w` may be 1-3), verify `windowed_scan`
     either falls back to `dense_scan` automatically or runs correctly
     with all planes treated as edge planes.

## Updated tests

1. **ducc parity tests**: re-run all v0.1 ducc parity tests with the
   new `n_w` and `windowed_scan` strategy. Must still pass `< 10 *
   eps`. If any regress, debug Part 1 before continuing.

2. **JAX integration tests**: jit, grad, vmap of `windowed_scan` —
   must all pass.

3. **Benchmark suite** (`tests/test_benchmark_against_ducc.py`):
   - Add `windowed_scan` and `windowed_vmap` rows.
   - Add an `n_pix` sweep `{128, 256, 512, 1024}` to characterise the
     FFT/spread crossover.
   - Add a `padding_overhead` column reporting `max_window_size /
     mean_window_size` per scenario.

## Memory tests

Peak RSS comparison across strategies. `windowed_scan` should stay
within `~1.2x` of `dense_scan` (slightly higher because of the sort
permutation tables and the per-plane gather buffers).

# Risks / open questions

1. **`x_0` empirical verification.** If ducc's `x_0` is not 0.25 or
   0.5 but something more nuanced (e.g., epsilon-dependent), Part 1
   needs a more careful match. If reading ducc source isn't enough,
   fall back to fitting `n_w` to ducc behavior empirically across the
   `(W, eps)` grid.

2. **`phi_hat` conditioning at large `W`.** If `W = 10+` produces
   parity failures even with bumped `phi_hat_oversample`, options
   are: (a) tighter safety floor, (b) different `beta` constant,
   (c) hybrid mode that uses v0.1's `x_0 = 1/W` only for `W >= 10`.
   Document whichever path is taken.

3. **Padding overhead for pathological w-distributions.** The
   diagnostic should warn when `max / mean > 3` and error or fall
   back automatically when `> 5`. The threshold values are guesses;
   refine after seeing benchmark numbers.

4. **`vis.at[rows].add(...)` performance on GPU.** The scatter-add
   must compile to an efficient XLA scatter. If it shows up hot in
   profiling, alternatives are `jax.ops.segment_sum` (requires sorted
   indices, which we have) or a custom XLA op. Defer measurement
   until GPU benches exist.

5. **GPU vs CPU strategy ranking.** All decisions about default
   `w_strategy` are based on CPU benchmarks. cuFINUFFT may favor
   different trade-offs; revisit defaults after the first GPU run.

6. **Boundary-plane edge cases.** When `n_w < W` (e.g., EDA2 with full
   sky and short baselines), every plane is an edge plane. Verify that
   this regime works correctly — it may be cleaner to fall back to
   `dense_scan` automatically when `n_w <= W`.
