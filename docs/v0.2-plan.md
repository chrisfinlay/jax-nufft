# v0.2 plan: windowed-per-plane visibility processing

## Goal

Close most of the v0.1-vs-ducc spreading-cost gap on CPU by pre-sorting
visibilities by `w_lambda` and giving each plane only its kernel-window of
visibilities. The target metric is the spread-work cost; FFT cost is
largely out of scope (see the "What this won't fix" section).

## Diagnosis (v0.1)

For each w-plane, v0.1 calls `jax_finufft.nufft2` over **all** `n_rows`
visibility points, then multiplies by the w-direction kernel
`phi((w_lambda - w_k) / scale)`. Because the kernel has compact support of
width `W * dw` in `w_lambda`, the multiplication zeroes out roughly
`1 - W/n_w` of the contributions. JAX's rigid-shape constraint forces us
to spread every visibility on every plane regardless.

For MWA-extended off-zenith with `n_w = 769`, `n_rows = 600`, and `W = 6`:

* v0.1 total spread work:  `n_w * n_rows * W^2 = 769 * 600 * 36 ~= 16.6M` spreads
* ducc total spread work:  `n_rows * W^3        = 600 * 216    ~= 130k` spreads

That's a ~130x ratio on spread alone. Most of the observed wall-clock gap
on small image sizes is dominated by FFT work (`n_w` 2D FFTs per call),
not spreading; but at production sizes (`n_pix >= 1k`) and tighter
epsilon (larger `W`) the spread term becomes significant, and the
windowed-per-plane scheme is what closes it.

## Algorithm: windowed per-plane scan

### Pre-sort

1. **Sort by `w_lambda`** at plan time (per channel). Store the
   permutation `sort_perm` and its inverse `inv_perm`. Visibilities with
   adjacent indices in the sorted array now have similar `w_lambda`.
2. **Per-plane window indices.** For each plane `k` at `w_k`, the
   visibilities that contribute (those with `|w_lambda - w_k| <= W * dw / 2`)
   form a contiguous range `[start_k, end_k)` in the sorted array. Cache
   `start_k` and the window size `size_k = end_k - start_k`.
3. **Worst-case padding.** Define `max_window_size = max_k size_k`. For a
   reasonably uniform `w_lambda` distribution this is typically
   `~1.5 - 2x` the mean.

### Forward operator

```python
for k in range(n_w):                              # lax.scan, n_w iters
    rows_k = jnp.dynamic_slice(sort_perm,         # shape (max_window_size,)
                               (start_k,),
                               (max_window_size,))
    u_k    = u_finufft[rows_k]
    v_k    = v_finufft[rows_k]
    w_k_lambda = uvw_lambda[..., 2][rows_k]
    mask_k = jnp.arange(max_window_size) < size_k

    image_k = image * exp(+2pi*i*w_centers[k]*nm1) / phi_hat_n
    contrib = nufft2(image_k, u_k, v_k, ...)      # NUFFT on max_window_size pts
    z       = (w_k_lambda - w_centers[k]) / scale
    contrib = contrib * phi(z, beta) * mask_k     # phi=0 for padding too

    vis = vis.at[rows_k].add(contrib)             # scatter, padding adds 0
```

### Adjoint operator

Mirrors the forward:

```python
for k in range(n_w):                              # lax.scan
    rows_k = jnp.dynamic_slice(sort_perm,
                               (start_k,),
                               (max_window_size,))
    vis_k  = vis[rows_k] * mask_k
    if weights is not None:
        vis_k *= weights[rows_k]
    z      = (w_k_lambda - w_centers[k]) / scale
    vis_k  = vis_k * phi(z, beta)

    H_k    = nufft1((n_l, n_m), vis_k, u_k, v_k, ...)
    dirty  = dirty + H_k * exp(-2pi*i*w_centers[k]*nm1) / phi_hat_n

return (dirty.real / n)
```

### Why this is the right shape for JAX

* The outer scan is over `n_w` planes — **same count as v0.1**, no FFT
  regression.
* `dynamic_slice` allows the per-plane window to start at a different
  index each iteration while keeping the *size* constant
  (`max_window_size`), so the NUFFT shape is rigid.
* The mask zeroes out padded rows; their contribution to vis (forward) is
  zero by construction, and any spread cost paid on padded rows is small
  relative to live rows since `max_window_size` is bounded.
* `vis.at[rows_k].add(contrib)` becomes an XLA scatter op. Because
  `phi(z) = 0` outside the kernel support, the padded rows in `rows_k`
  are still in-bounds indices but their `contrib` is zero, so the
  scatter is harmless.

## Plan-side additions to `WGridderPlan`

| field             | shape                       | contents                                                |
|-------------------|-----------------------------|---------------------------------------------------------|
| `max_window_size` | `int` (static)              | worst-case visibility count in any plane's kernel window |
| `sort_perm`       | `(n_chan, n_rows)` int      | permutation that sorts uvw_lambda by w within channel    |
| `window_start`    | `(n_chan, n_w)` int         | start index in sorted array for plane `k`'s window       |
| `window_size`     | `(n_chan, n_w)` int         | live (unpadded) length of plane `k`'s window             |

`make_plan` also surfaces a diagnostic
`window_padding_overhead = max_window_size / mean_window_size` so callers
can detect pathological uvw distributions.

## Cost analysis

Let `mean_window_size = n_rows * W / n_w` (each visibility is in the
kernel support of `~W` planes; total plane-touches = `n_rows * W`,
distributed across `n_w` planes).

### Spread

* v0.1:    `n_w * n_rows * W^2`
* v0.2:    `n_w * max_window_size * W^2`
        `~= n_w * (p * mean_window_size) * W^2`
        `~= p * n_rows * W^3`     (where `p ~ 2` is padding overhead)
* ducc:    `n_rows * W^3`

So v0.2 spread cost is `~2x` ducc and `~(n_w/(2W))` lower than v0.1.

### FFT

* v0.1:  `n_w` 2D FFTs per call (one per plane, image-size FFT each)
* v0.2:  `n_w` 2D FFTs per call (**unchanged**)

This is the key advantage of the windowed scheme over the
bucket-by-w-range alternative discussed below: the outer loop stays at
`n_w`, so FFT count doesn't grow.

### Net effect by problem regime

| Regime                      | FFT cost   | Spread cost   | Windowed benefit              |
|-----------------------------|------------|---------------|-------------------------------|
| Small images (n_pix <= 256) | dominant   | small         | small but positive (no FFT regression) |
| Medium (n_pix ~ 512)        | comparable | comparable    | meaningful                     |
| Large (n_pix >= 1024)       | small      | dominant      | substantial win               |
| Tighter eps (W >= 9)        | unchanged  | grows W^3     | substantial win               |
| GPU                         | well-batched | well-batched | the scatter-add and the mask are extra ops a GPU can hide; bucketed scheme below may pull ahead due to better launch coalescing |

## Alternative: bucket-keyed scan (worse on CPU; might still help on GPU)

A different formulation pre-sorts visibilities and partitions them into
`B` rigid-size buckets each spanning ~`M * dw` in w-space, then iterates
over buckets in the outer scan and over the `~M + W` planes that overlap
each bucket in the inner scan.

The trouble is that **a plane near a bucket boundary is reached by two
buckets** (its kernel `[w_k - W*dw/2, w_k + W*dw/2]` extends across the
boundary), so each plane fires multiple NUFFT calls. Concretely, with
buckets of width `M * dw`:

* `B = n_w / M`
* `planes_per_bucket = M + W`  (plane centres in the bucket *plus* W/2
  on each side from neighbours whose kernels reach in)
* Total NUFFTs = `B * planes_per_bucket = n_w * (1 + W / M)`

For `M = W` (kernel-width buckets): **2 * n_w NUFFTs**, double the
windowed scheme. The spread *cost* per call is comparable to the
windowed scheme, so the doubled NUFFT count translates directly into
doubled FFT work. On CPU this is a regression; on GPU the kernel-launch
coalescing might offset the cost — worth benchmarking but not the
default.

The bucket-keyed scheme would be the right choice if (and only if):

* the per-launch overhead in cuFINUFFT is large enough that fewer-but-larger
  batched calls beat more-but-smaller calls, **and**
* the doubled FFT work doesn't dominate.

We deliberately don't commit to bucketing as the default; it's a v0.3+
question once we have GPU numbers in hand.

## What this won't fix

* **n_w 2D FFTs per call.** Each plane needs its own image (different
  phase shift), so the FFT count is fundamentally tied to `n_w`. Reducing
  it requires a different decomposition (e.g. faceting, or shifting the
  per-plane phase to a Fourier-domain convolution). Out of scope for v0.2.
* **Off-zenith long-baseline imaging at large `n_pix`.** Even at v0.2's
  best, the FFT cost dominates and we'll still pay `n_w` FFTs. ducc's
  hand-fused spread+FFT path will stay faster on CPU.

## Test additions

* Unit tests for the window builder:
  - `window_size` summed over planes equals `n_rows * W` (each visibility
    is in W kernel windows on average — modulo edge effects),
  - `window_start[k]` is monotonic in `k` (because we sorted by `w`),
  - sentinel slots produce zero contribution (mask check).
* Parity tests cross-checking the windowed forward / adjoint against the
  existing `scan` strategy at all four telescopes x both pointings x
  `eps` in `{1e-4, 1e-6}`.
* Bench: extend `tests/test_benchmark_against_ducc.py` with a third
  jax-side strategy `windowed`, alongside `scan` and `vmap`. Add a sweep
  over `n_pix in {128, 256, 512, 1024}` to characterise the FFT/spread
  cross-over.
* Memory: peak-RSS comparison; `windowed` should stay close to `scan`'s
  bounded footprint.

## Implementation order

1. **Plan**. Add `sort_perm`, `window_start`, `window_size`, and
   `max_window_size` to `WGridderPlan`. Update `make_plan`. Unit-test
   the window builder. Surface `window_padding_overhead` diagnostic.
2. **Forward**. Implement `dirty2vis` with `w_strategy="windowed"`.
   Cross-check parity with `scan` on every telescope/pointing combination.
3. **Adjoint**. Same.
4. **Strategy registration**. Add `w_strategy="windowed"` to the public
   API. Decide whether it becomes the new default (likely yes, since
   it's strictly better than `scan` in every regime tested).
5. **Bench**. Add `windowed` to time + memory bench. Run the n_pix
   sweep. Update the README tables.
6. **Optional**: implement the bucket-keyed scheme as a second new
   strategy (`w_strategy="bucketed"`) and benchmark on GPU. If it
   doesn't pay off, drop it.
7. **Tag** v0.2.0.

## Risks / open questions

* **Padding overhead is data-dependent.** For very compact arrays, the
  density of visibilities in the smallest-w region might be much higher
  than the density in the tails, inflating `max_window_size` well above
  the mean. The diagnostic should warn when `max / mean > 3`.
* **Edge planes.** The first and last `~W/2` planes have only partial
  kernel support extending beyond the data range. Handle by extending
  `window_start` past the live range and letting padding do its work.
  Unit-test target.
* **Multi-channel.** `uvw_lambda` is per-channel so the sort and the
  per-plane window indices are per-channel. Memory for the index tables
  is `n_chan * n_w * 8 bytes`, which is tiny.
* **`vis.at[rows].add(...)` performance.** Scatter-add into a large
  array is fine in XLA but the cost should be measured. An alternative
  is `jax.ops.segment_sum` if the scatter shows up hot.
* **GPU vs CPU may flip the scheme ranking.** cuFINUFFT amortises
  kernel-launch overhead via batched calls, so the bucketed alternative
  might reclaim its cost on GPU. Both schemes need GPU benchmarking
  before we commit defaults.

## Out of scope for v0.2

* Polarisation.
* Phase-center offset.
* Faceting.
* BDA / smearing correction.
* Per-visibility frequency.
* Replacing the `n_w` 2D FFTs with anything fewer (would require a
  different algorithm).
