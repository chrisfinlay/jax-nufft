# v0.2 plan: bucketed visibility processing

## Goal

Close most of the v0.1-vs-ducc spreading-cost gap on CPU by pre-sorting
visibilities and processing them in fixed-size buckets so that each NUFFT
call sees only the visibilities that actually contribute to it. The
target metric is the spread-work cost; FFT cost is largely out of scope
(see the "What this won't fix" section).

## Diagnosis (v0.1)

For each w-plane, v0.1 calls `jax_finufft.nufft2` over **all** `n_rows`
visibility points, then multiplies by the w-direction kernel
`phi((w_lambda - w_k) / scale)`. Because the kernel has compact support of
width `W * dw` in `w_lambda`, the multiplication zeroes out roughly
`1 - W/n_w` of the contributions. JAX's rigid-shape constraint forces us
to spread every visibility on every plane regardless.

For MWA-extended off-zenith with `n_w = 769`, `n_rows = 600`, and `W = 6`:

* v0.1 total spread work:  `n_w * n_rows * W^2 = 769 * 600 * 36 ≈ 16.6M` spreads
* ducc total spread work:  `n_rows * W^3        = 600 * 216    ≈ 130k` spreads

That's a ~130x ratio on spread alone. Most of the observed wall-clock gap
on small image sizes is in fact dominated by FFT work (n_w 2D FFTs per call),
not spreading; but at production sizes (n_pix >= 1k) and tighter epsilon
(larger W) the spread term becomes significant, and bucketing is what
closes it.

## Algorithm

### Pre-sort and bucket

1. **Sort by `w_lambda`** at plan time (per channel). Store the permutation.
2. **Partition** the sorted visibilities into `B` buckets, each spanning a
   w-range of roughly `M * dw` for some integer `M >= W`. The natural
   choice is `M = W`, giving buckets one kernel-width wide.
3. **Pad** each bucket to a fixed `max_bucket_size` with sentinel rows.
   Padding overhead is bounded by the max/mean bucket-size ratio, which
   for sensible uvw distributions is typically 1.5-2x.
4. **Map planes to buckets**: for each bucket `b` (centred at `w_b_centre`,
   spanning `w_b_lo` to `w_b_hi`), enumerate the ~`2W` planes whose
   kernels overlap the bucket — concretely, planes with
   `|w_k - w_b_centre| <= W * dw / 2 + (w_b_hi - w_b_lo) / 2`.

### Forward operator

```python
for b in range(n_buckets):                      # lax.scan, B iterations
    rows_b   = sort_perm[bucket_indices[b]]     # shape (max_bucket_size,)
    u_b, v_b = u_finufft[rows_b], v_finufft[rows_b]
    w_b      = uvw_lambda[..., 2][rows_b]
    mask_b   = bucket_mask[b]                   # bool, padding -> False
    vis_b    = jnp.zeros((max_bucket_size,), complex)

    for k in bucket_planes[b]:                  # inner lax.scan, ~2W iters
        image_k    = image * exp(+2pi*i*w_centers[k]*nm1) / phi_hat_n
        contrib    = nufft2(image_k, u_b, v_b, ...)
        z          = (w_b - w_centers[k]) / scale
        vis_b     += contrib * phi(z, beta)     # phi=0 outside support

    vis_b = jnp.where(mask_b, vis_b, 0.0)
    vis   = vis.at[rows_b].add(vis_b)           # scatter
```

### Adjoint operator

Mirrors the forward with `nufft1` and image-domain accumulation:

```python
for b in range(n_buckets):                      # lax.scan
    rows_b      = sort_perm[bucket_indices[b]]
    vis_b       = vis[rows_b] * mask_b
    if weights is not None:
        vis_b  *= weights[rows_b]
    u_b, v_b, w_b = ...
    dirty_b     = jnp.zeros((n_l, n_m), complex)

    for k in bucket_planes[b]:                  # inner lax.scan
        z       = (w_b - w_centers[k]) / scale
        vis_kb  = vis_b * phi(z, beta)
        H_k     = nufft1((n_l, n_m), vis_kb, u_b, v_b, ...)
        dirty_b = dirty_b + H_k * exp(-2pi*i*w_centers[k]*nm1) / phi_hat_n

    dirty = dirty + dirty_b

return (dirty.real / n)
```

A given plane's image-domain phase shift is computed multiple times across
buckets in this scheme (since each plane belongs to ~2 buckets when
`M = W`). Pre-computing the phase shift once per plane and storing the
`(n_w, n_l, n_m)` stack would be the alternative, but that's the same
memory blowup as `w_strategy="vmap"` today; recomputing keeps memory
bounded at `O(n_l * n_m)` per scan iteration.

## Plan-side additions to `WGridderPlan`

| field                | shape                                    | contents                                              |
|----------------------|------------------------------------------|-------------------------------------------------------|
| `n_buckets`          | `int` (static)                           | number of buckets                                     |
| `max_bucket_size`    | `int` (static)                           | worst-case padded visibility count per bucket         |
| `planes_per_bucket`  | `int` (static)                           | constant ~`2W` (kernel overlap)                       |
| `sort_perm`          | `(n_chan, n_rows)` int                   | permutation that sorts uvw_lambda by w within channel |
| `bucket_indices`     | `(n_chan, n_buckets, max_bucket_size)`   | sorted-vis indices per bucket; padded with sentinel   |
| `bucket_mask`        | `(n_chan, n_buckets, max_bucket_size)`   | True for live slots, False for padding                |
| `bucket_planes`      | `(n_buckets, planes_per_bucket)`         | plane indices for each bucket; padded with sentinel   |

`make_plan` also surfaces a diagnostic `padding_overhead = max_bucket_size /
mean_bucket_size` so callers can detect pathological uvw distributions.

## Cost analysis

Let `M` be the bucket width in plane-spacings. The relevant counts are:

* `B = n_w / M`
* `planes_per_bucket ≈ M + W`
* `mean_bucket_size = n_rows / B`

### Spread

* v0.1:   `n_w * n_rows * W^2`
* v0.2:   `B * planes_per_bucket * max_bucket_size * W^2`
        `≈ (n_w / M) * (M + W) * (2 * n_rows / B) * W^2`
        `≈ 2 * (M + W) * n_rows * W^2`

For `M = W`: v0.2 spread ≈ `4 * n_rows * W^3`. ducc is `n_rows * W^3`. So
v0.2 lands within ~4x of ducc on spread, ~32x lower than v0.1 (for
`n_w / W ≈ 130`).

### FFT

* v0.1:   `n_w` 2D FFTs per call (one per plane, image_size FFT each)
* v0.2:   `B * planes_per_bucket = n_w * (1 + W/M)` 2D FFTs per call

For `M = W`, FFT count is **2x** v0.1. There's no way around this with
bucketing alone: each plane needs its own image (different phase shift),
and that requires its own FFT.

### Net effect by problem regime

| Regime                      | FFT cost | Spread cost | Bucketing benefit |
|-----------------------------|----------|-------------|-------------------|
| Small images (n_pix <= 256) | dominant | small       | likely a regression (2x FFT, small spread savings) |
| Medium (n_pix ~ 512)        | comparable | comparable | break-even       |
| Large (n_pix >= 1024)       | small  | dominant      | substantial win  |
| Tighter eps (W >= 9)        | unchanged | grows W^3   | substantial win  |
| GPU                         | well-batched | well-batched | larger constant-factor wins from cuFINUFFT batch launches |

So bucketing is most attractive for production-sized problems and tighter
accuracy targets, not the small benchmark configs we currently exercise.

## Alternative: windowed per-plane scan

If bucketing turns out to be a net loss in our target regime, the
strictly-better-than-v0.1 alternative is:

1. Pre-sort by w (same as bucketing).
2. Compute, per plane, the contiguous range of sorted-visibility indices
   whose w-coords fall inside the kernel support `[w_k - W*dw/2,
   w_k + W*dw/2]`.
3. Pad every plane to `max_window_size` (the worst-case range size).
4. `lax.scan` over **`n_w` planes** as in v0.1, but each iteration's
   NUFFT operates on `max_window_size` rows instead of `n_rows`.

This keeps the FFT count at `n_w` (no regression) while bringing spread
cost down to `n_w * max_window_size * W^2 ≈ 2 * n_rows * W^3` — within
~2x of ducc, and strictly better than v0.1.

The bucketed scheme is asymptotically cleaner (it generalises to
strategies that share work across multiple planes within a bucket), but
the windowed scheme is the cheaper engineering target if we just want
"v0.1 but with ducc-comparable spread cost."

Recommendation: implement and benchmark both; pick whichever wins on the
production-sized problem.

## What this won't fix

* **n_w 2D FFTs per call.** Each plane needs its own image (different
  phase shift), so the FFT count is fundamentally tied to `n_w`. Reducing
  it requires a different decomposition (e.g. faceting, or shifting the
  per-plane phase to a Fourier-domain convolution). Out of scope for v0.2.
* **Off-zenith long-baseline imaging at large n_pix.** Even at v0.2's
  best, the FFT cost dominates and we'll still pay `n_w` FFTs. ducc's
  hand-fused spread+FFT path will stay faster on CPU.

## Test additions

* Unit tests for the bucket builder:
  - bucket counts match `ceil(w_extent / (M * dw))`,
  - bucket_mask sums to `n_rows` per channel,
  - every visibility appears in exactly one bucket,
  - sentinel slots produce zero contribution.
* Parity tests cross-checking the bucketed forward / adjoint against the
  existing scan strategy at all four telescopes x both pointings x
  eps in {1e-4, 1e-6}.
* Bench: extend `tests/test_benchmark_against_ducc.py` with a third
  jax-side strategy `bucketed`, alongside `scan` and `vmap`.
* Memory: peak-RSS comparison; bucketed should sit between scan and vmap
  (one bucket's worth of working set, vs scan's one plane's worth).

## Implementation order

1. **Plan**. Add bucket arrays to `WGridderPlan` and update `make_plan`.
   Unit-test the bucket builder. Surface `padding_overhead` diagnostic.
2. **Forward**. Implement bucketed `dirty2vis`. Cross-check parity with
   `scan` strategy on every telescope/pointing combination.
3. **Adjoint**. Same.
4. **Strategy registration**. Add `w_strategy="bucketed"` (and likely
   `"windowed"` for the alternative). Decide which becomes the default
   based on bench results.
5. **Bench**. Add `bucketed` to time + memory bench. Run a sweep over
   `n_pix in {128, 256, 512, 1024}` to characterise when bucketing wins.
6. **README**. Update strategy comparison and bench tables.
7. **Tag** v0.2.0.

## Risks / open questions

* **Padding overhead is data-dependent.** Realistic uvw distributions
  might have very lopsided bucket sizes if the array is very compact,
  inflating `max_bucket_size`. Need the diagnostic to warn when
  `max / mean > 3`.
* **Edge buckets / planes.** The first and last `~W/2` planes have only
  partial kernel support; the same applies to the first/last bucket.
  The bucket-to-plane overlap table needs to handle these correctly.
  This is a unit-test target.
* **Multi-channel.** v0.1 already runs a per-channel scan because
  uvw_lambda is per-channel. The bucket layout depends on per-channel
  w-distribution, so each channel needs its own `bucket_indices`. Memory
  for the index tables is `n_chan * n_buckets * max_bucket_size * 8 bytes`,
  which can grow but is much smaller than the image stack vmap allocates.
* **GPU vs CPU may flip the analysis.** cuFINUFFT amortises kernel-launch
  overhead via batched calls, making the doubled FFT count cheaper. The
  bucketing benefit is also larger on GPU because spread parallelism is
  the limit there. Both strategies need GPU benchmarking.
* **`vis.at[rows].add(vis_b)` performance.** Scatter-add into a large
  array is fine in XLA but the cost should be measured. An alternative
  is to use `jnp.zeros((n_rows + 1,))` with sentinel index `n_rows` and
  drop the last element after the scan.

## Out of scope for v0.2

* Polarisation.
* Phase-center offset.
* Faceting.
* BDA / smearing correction.
* Per-visibility frequency.
* Replacing the n_w 2D FFTs with anything fewer (would require a
  different algorithm).
